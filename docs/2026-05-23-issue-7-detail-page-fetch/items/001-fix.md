## Nits captured for follow-up (not blocking this PR)

The 3 nits surfaced by review + pr-review are documentation/design concerns; none would
produce a wrong output or crash. Recording them here so they're searchable for the
next time someone touches this code, but they will NOT block merge:

1. **Docstring on `test_crawl_with_detail_falls_back_on_detail_404`** — the literal
   test docstring says "Crawl still succeeds with listing-only data" but the
   implementation actually halts on 404 → PartialHarvest. The assertions
   (`result.status == OK`, `len(result.jobs) == 1`, `company is None`) hold because
   PartialHarvest reports `status=OK`. Suggested rewrite: *"404 on a detail page
   halts further enrichment and converts the run to PartialHarvest, preserving
   prior listing-only data."* — defer to a follow-up commit since it doesn't
   affect behaviour or test coverage.

2. **404 = halt-the-whole-crawl semantics** — per the plan and CONTEXT.md
   PartialHarvest entry, classify-non-OK on any detail page halts further enrichment.
   In real-world TesterHome data, individual topic 404s (deleted threads) are routine,
   so this may be aggressive. Worth a follow-up issue to consider per-job degrade for
   plain `404` (text-body) vs. PartialHarvest for `429`/`5xx`/soft-blocks. Out of
   scope for this PR.

3. **Cache-hit re-classification uses empty `headers={}`** — safe today because
   `classify()` only reads `headers` for `Retry-After` on 429s, and cache hits are
   always 200. If the cache layer is ever extended to surface response headers,
   this assumption needs revisiting. Doc-only nit; no code change.
