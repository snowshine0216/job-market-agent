job-market-agent · Current Implementation Workflow

Phase 1 single-source slice — TesterHome listings → SQLite. Source owns fetch (callbacks for cache + blobs); canonical_id (ADR-0001) + URL-freshness merge (ADR-0003).

End-to-end flow (numbered above)

  • • 1. cli.py loads config/sources/testerhome.yaml and builds a source factory
  • • 2. pipeline.crawl.run opens DB, calls start_run, invokes source.crawl()
  • • 3. Per page inside the source: cache_get → on miss http.fetch → on_fetch writes url_cache + gzip blob
  • • 4. blockage.classify → ok / blocked / rate_limited / error (returns BlockStatus)
  • • 5. selectolax extracts items → normalize.parse_* for salary / location / experience
  • • 6. dedup.job_id (source-scoped) + dedup.canonical_id (cross-source merge key, ADR-0001)
  • • 7. Source returns SourceResult → insert_jobs (ON CONFLICT update; url_status conditional, ADR-0003) → finish_run always (ADR-0002)

Boundaries & invariants

  • • domain/* is pure: deterministic, no network, no mutation
  • • http.py is a pure transport — classify / parse / blobs are called from the source, not from http.py
  • • pipeline/crawl.py owns the lifecycle; the source owns fetch (factory-with-callbacks)
  • • Pydantic v2 models are frozen — always model_copy(update=…)
  • • finish_run runs on every code path (even on exception) — no dangling runs
  • • Aggregations summarising Jobs (not JobObservations) must GROUP BY canonical_id

What's NOT in this slice yet

  • • Randstad, Bing, Playwright sources (Phase 2)
  • • LLM client, extraction, narration (Phase 3+)
  • • Skills canonicalisation / skills.yaml (Phase 3)
  • • Market & fit reports, persona, scoring (Phase 4–5)
  • • jma run, sources status, cross-run delta (Phase 2/6)
